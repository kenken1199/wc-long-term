================================================================================
  工程能力指数として Pp/Ppk のみを表示する理由
  （本ツールにおける設計判断の記録）
================================================================================
作成日: 2026-05-20

────────────────────────────────────────────────────────────────────────────────
1. Cp/Cpk と Pp/Ppk の違い
────────────────────────────────────────────────────────────────────────────────

式の形は同じ。違いは使うσ（標準偏差）だけ。

  Pp  = reject_limit / (3 × σ_overall)
  Ppk = (reject_limit − |平均 − 公称重量|) / (3 × σ_overall)

  Cp  = reject_limit / (3 × σ_within)
  Cpk = (reject_limit − |平均 − 公称重量|) / (3 × σ_within)

  σ_overall … 全OKデータをまとめてstd()で計算。ロット間変動を含む。
  σ_within  … ロット内のばらつきのみ。ロット間のシフトを除外。

Cp/Cpk が「潜在能力」、Pp/Ppk が「実績性能」と呼ばれるのは、
Cp/Cpk がロット間変動を無視した楽観的な数字になりやすいため。


────────────────────────────────────────────────────────────────────────────────
2. 1ロット単位での Cpk は意味が薄い
────────────────────────────────────────────────────────────────────────────────

Cpk には「工程が統計的管理状態にある」という前提がある。
1ロットのデータだけでは以下が確認できない。

  ・原料ロット差
  ・温湿度・季節による変動
  ・設備の経時劣化
  ・作業者・シフトの違い

→ 1ロットだけの Cpk は「そのロット内の参考値」に過ぎず、
  工程能力の評価には使えない。

また本ツールでは基準値（規格の中心）を「最初の5個の平均」としている。
これはロットごとに変わるため、ロット間でCpkを比較することも無意味になる。

【結論】単ロット Cpk は実装しない。


────────────────────────────────────────────────────────────────────────────────
3. 複数ロット集計で Cp/Cpk を出さない理由
────────────────────────────────────────────────────────────────────────────────

Cp/Cpk には σ_within（ロット内σ）が必要。
σ_within を正確に求めるにはロット境界を正しく特定しなければならない。

本ツールが使う時間ギャップ法（30分以上の空きでロット区切り）は信頼できない。

  問題① : 30分以内でロット切替している場合
           → 2ロットを1ロットと誤判定 → σ_within が実態より大きくなる

  問題② : 40分休憩して同じロットを続けている場合
           → 1ロットを2ロットと誤判定 → σ_within が実態より小さくなる

  問題③ : 昼休み・夜間停止など
           → ほぼ確実に誤判定が発生する

1日に同じ製品を3ロット製造するケースが実際に存在することも確認済み。
ファイル（日付フォルダ単位）でσを集計する現在の構造では、
1日3ロット分のロット間シフトがσ_withinに混入してしまう。

代替手段として「移動範囲法（MR法）」も検討したが、
ロット境界をまたぐ隣接ペアを除外する処理が必要になり、
その除外自体がまた時間ギャップに依存するという循環問題が残る。

【結論】信頼できるロット境界の自動特定が不可能なため、
        σ_within を正確に計算できない → Cp/Cpk は出さない。


────────────────────────────────────────────────────────────────────────────────
4. Pp/Ppk が適切な理由
────────────────────────────────────────────────────────────────────────────────

Pp/Ppk は σ_overall（全データの標準偏差）を使う。
これはロット境界に依存しない。

  ・全OKデータをまとめてstd()するだけ → ロット境界の特定が不要
  ・ロット内変動 + ロット間変動 + 長期ドリフト、すべてを含む
  ・「実際にお客様に届く製品のばらつき」を正直に表す数字

複数ロット・複数日のデータが蓄積されるほど安定する。
一般的な目安：

  1ロット    → 参考値レベル（そのロットだけの実績）
  5〜10ロット → 暫定評価に使える
  20〜30ロット → 工程の真の実力として信頼できる

本ツールの record 集計（複数ロット・複数日の集計）で算出するのは
まさに 20〜30ロット以上のデータを前提とするため、Pp/Ppk が最適。


────────────────────────────────────────────────────────────────────────────────
5. 規格値（USL/LSL）について
────────────────────────────────────────────────────────────────────────────────

本来 Pp/Ppk の計算には「顧客との取り決めた規格値」が必要だが、
本製品には外部規格値がない。

代わりに「排除上下限（reject_limit）」を規格として使用する。
  例: 基準値 ± 1.2g、± 2.0g など（製品ごとに config.yaml に設定）

この解釈の意味：
  Pp ≥ 1.33 → 3σ範囲が排除限界の中に十分収まっている
             → 機械が排除しているNG品は外れ値であり、工程は安定
  Pp < 1.00 → 3σ範囲が排除限界を超えている
             → 理論上、排除限界内に収まらない製品が一定数発生

Ppk については公称重量（nominal_weight）が config.yaml に登録されている
製品のみ計算する。未登録の場合は Pp のみ表示。


────────────────────────────────────────────────────────────────────────────────
6. まとめ
────────────────────────────────────────────────────────────────────────────────

  表示する指標  : Pp / Ppk のみ
  表示しない指標: Cp / Cpk（ロット境界の自動特定が不可能なため）

  計算箇所     : record 集計の「複数ロット・複数日」画面のみ
                 （単ロット分析では出さない）

  使うσ        : σ_overall（全OKデータの標準偏差）
  使う規格      : 製品ごとの排除上下限（reject_limit_g）
  公称重量あり  : Pp と Ppk を表示
  公称重量なし  : Pp のみ表示（Ppk は "-（公称重量未登録）"）

  判定目安      : Pp/Ppk ≥ 1.33 を推奨（一般的な品質管理の合格基準）
